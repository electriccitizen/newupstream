Sass

Citizen Patterns uses Sass as our CSS preprocessor of choice. CSS Preprocessors allow a themer to write theming code using a specific syntax which the preprocessor then turns into browser-friendly CSS. Preprocessors help you write more efficient re-usable code with less repetition than regular CSS. They also allow the use of variables, functions and expressions not allowed in regular CSS.

Citizen Patterns uses Scss (Sassy CSS) files to hold its Sass markup. Scss is the main syntax of Sass as of version 3. Citizen Patterns do not use any Sass frameworks (i.e. Compass, Bourbon, Susy) but rather uses a multitude of mixins based on site needs. This allows for a lighter weight theme and more personal developer customization.

Citizen Pattern SCSS Files

Virtually every component in OC Patterns has a Scss file. Some have high level Scss files for the component level and then more specific Scss files for the component itself (example: _widgets.scss and _slideshow.scss). All Scss files should have a preceding underscore (_example.scss to hide them from Pattern Lab. The CSS sheet they get compiled to is visible to Pattern Lab so the individual Scss files should not be.

Mixins & Placeholders

A lot of the power of Sass comes from mixins and placeholders. These are bits of reusable code that prevent you from having to type the same thing over and over. Mixins are denoted as @mixin example{ my scss code here } and called with includes: @include example;. Placeholder are denoted as %example{ my scss code here} and called with extends: @extend %example.

The actual difference between Mixins and Placeholders can be tricky but here are a couple of key points:

  1. Variable arguments cannot be passed through placeholders.
  2. Placeholders must be used in the file they are declared in, or that file must be imported into the file they will be used in.
  3. Mixins are printed for each selector instance, but place holders are printed once with the selectors using them grouped together.
The top two key differences definitely make the argument for using nothing but mixins, however, as noted in the last key difference they do slightly inflate the size of your stylesheet versus placeholders.

Citizen Patterns is generally favors mixins over placeholders because of the allowance of variable arguments, but placeholders are definitely recommended for static Scss mixins with no contextual needs.

Citizen Mixins

Most mixins are defined in 00-base/_02-mixins.scss. Background mixins are defined in 00-base/_03_background-sizes.scss and breakpoint mixins are set in 00-base/_00-breakpoints.scss. Some other more specific mixins are set in their respective component Scss files (example: @mixin blockTitle{} set in 01-atoms/02-text/00-headings/_headings.scss).

All mixins set in the theme should be listed as a mixin-tile in 00-base/global/04-mixins/mixins.twig so that there is a visual Pattern Lab reference for them. The mixin tile should give the mixin name, the @include call for it, a short description, any argument values needed, and a list of similar mixins (similar mixins do not need to have a tile for each).

A new mixin should be created anytime a piece of Scss is being reused in multiple files of the theme and there are contextual variables being set in each instance.
Example:
@mixin columns($columns, $gap){
-webkit-column-count: #{$columns};
-moz-column-count: #{$columns};
column-count: #{$columns};
-webkit-column-gap: #{$gap};
-moz-column-gap: #{$gap};
column-gap: #{$gap};
}

Citizen Placeholders

Most placeholders are defined in 00-base/_01-placeholders.scss (which is imported into other Scss files as needed. File specific placeholders can be defined within the file they are being used in.

A placeholder should be created for any common piece of Scss that is reused across the site (or within a single file) but no contextual arguments are needed.
Example:
%listReset{
list-style:none;
padding: 0;
margin: 0;
}

Sass Linting

Sass linting is a way to force good code standards and consistency on Scss files as well as checking for common mistakes that will break the CSS when compiled. Learn more about general Sass linting in this great Electric Citizen blog post.

Citizen Patterns uses some minimal Sass linting run by StyleLint through the gulp clean task run on npm start. Linting rules are defined in the theme root .stylelintrc file.

The default severity for rules is set to warning since none of the current rules is serious enough to prevent compiling. These warnings will show in the command line during the Scss compiling and will give the file name the infraction is in followed by line number and column number of the error, and a description of the rule being broken.

Current Citizen Pattern Specific Linting rules:

  • Indentation: tabs
  • Allowed @ rules: include, mixin, extend, if, each, content, for, else.

Besides these specific Citizen Patterns lint rules, the gulp theme task run from npm start will check for basic Scss syntax errors (example: margin-bottom: 20px: ) that will break the theme when compiled. These errors will show in red in the command line and will prevent the theme from compiling until fixed.

Citizen Patterns Sass Best Practices & Conventions

  1. Comments— File names are commented at the start of Scss files.
    /*===================
    MY GREAT FILE NAME
    ========================*/

    Major sections/groups of Scss code are commented within the file.
    /* --------------------------
    Section/Group name
    --------------------------*/

    Single Scss elements withing sections are commented within each section.
    /* Element Name */

    Descriptive comments are added within each element above what they are describing.
    // my great descriptive comment

    Closing comments are added to Scss element closing brackets using a description comment on the same line as the closing bracket.
    }//end .my-selector

    Closing comments are also used when nest levels get more than 2 or 3 deep or it is not easy tell what a nested closing bracket pertains to due to the amount of nested code.

  2. Nesting— Top level selectors generally pertain to the component of each Scss file. Example: In 03-organisms/00-paragraphs/button/_button.scss all Scss code is nested within the .paragraph--type--buttons{} selector. because that Scss file pertains to that specific component.

    For files with multiple components (like _widgets.scss or _views.scss), top level selectors would generally be each main component in that file (each view instance would be its own top level selector).

    Selectors should never be nested more than 5-6 levels deep and 2-4 is usually preferred for clarity. Example: A view selector would have the view content, views row, and each view field all as second level nested elements—rather than a view selector with view content nested then views row nested inside that, with view fields nested inside the row. Using shallower nesting techniques makes the view easier to read at a glance and allows for clearer and more specific breakpoint nesting.

  3. Breakpoint nesting— Citizen Patterns uses breakpoint mixins based on grid variables set in 00-base/00-breakpoints.scss for responsive theming. Each individual selector (nested or not) should have its breakpoint styles nested directly in the selector. Example: .views-row{ default styles for the row; @include tab{ tablet styles for the row;} } and .my-viewfield{ default styles for the field; @include tab{ my tablet styles for the field;} }.

    Nesting breakpoint theming directly in the selector it pertains two provides an easier to read Scss file and limits the amount of selector repetition. Breakpoint theming should be at the end of each selector and should go in break point order: default, tab, desk, contentStop, etc. Special breakpoints can also be used to limit theming to just phones (less than 500px) or just mobile menu widths for example.

  4. Grid theming— Citizen Patterns does not use any defined grid system like Singularity or CSS Grids. This is because needs of the design and the build does not often make a single simple system worthwhile since things move position between mobile and desktop or are grouped in complex ways. Instead we use flex theming whenever possible and floats if absolutely necessary.

    Whenever possible use flex theming with the custom flex mixins defined in 00-base/_02-mixins. @include safeFlex($direction,$wrap,$justify,$align); will provide all necessary arguments for any type of flex theming you need for layout elements and will ensure it works back to IE10. @include safeFlexOrder($order) allows you to set the flex order of a child element with a flex layout and @include safeFlexSize($grow,$shrink,$size) lets you set flex sizing on the child element (regular percentage, vw, or pixel sizing is usually used instead of safeFlexSize however).

  5. Mixin, Mixin, Mixin— Use mixins and placeholders whenever you find yourself repeating the same Scss code in more than a few spots. Mixins with arguments can definitely help your efficiency while placeholders help keep the CSS lean. View defined mixins and placeholders in the Base > Global > Mixins Reference sheet. Whenever you create a new mixin, make sure you add a mixin tile for it in 00-base/04-mixins/mixins.twig so that it because visible in the mixins reference sheet.

    In order for a mixin to be used in a file, the mixin must be assigned in the file hierarchy before it is called in a file. This is why the 01-mixins.scss is one of the first top level files in the theme. Any file that uses placeholders that are defined in 01-placeholders.scss must import the 01-placeholders.scss file.

  6. Variables— All colors, grid sizes and fonts should be assigned to a variable. Mixins should take advantage of argument variables whenever their is a contextual need (like setting a different height for the vertRule mixin in different uses). If a mixin does not use an argument, it should probably be a placeholder.
  7. Camel Casing— When creating mixins, placeholders, or variables, always use camelCasing instead of hyphens or underscores.
