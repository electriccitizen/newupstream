Custom Twig Templates

Every component in Drupal has a corresponding Twig template or even group of templates. Whenever you need to change or create the HTML structure of a component, need to attach a JS file to it for scripting, or want to add some special classes for theming, you will need to create a new custom template for the element or edit an existing template.

Existing Custom Templates— All existing Drupal custom templates for Citizen Patterns are stored in /templates/zzz-custom. Any other templates in the theme are default templates from the Stable base theme. If any of these templates need to be edited—it should be moved into one of the sub-folders in the zzz-custom folder. Make sure you move the whole template instead of just copying a version of it into the zzz-custom folder structure. NOTE: You may also inherit templates from outside zzz-custom into templates inside of zzz-custom since it is at the end of the /template file hierarchy.

Existing Pattern Lab templates are all in the /components/_patterns file structure. Any new templates added to Pattern Lab should be added there in the appropriate component area. If the new Pattern Lab template is going to be inherited into Drupal, a custom template must also be created (or edited) for the inheritance in /templates/zzz-custom.

Any Twig template found outside of the Citizen Patterns theme is either a Drupal core template or one provided by a contributed module. Whenever possible, it is best not to customize these templates because this may cause errors after future core/module updates, but if it is necessary to do so, you should copy the template (not move it like with customizing Citizen Pattern default templates) into the zzz-custom folder structure.

Whenever you create a new template or rename a template, you must clear cache in order for the new template to be read by Drupal or Pattern Lab. Often when you make changes to a template, you may also need to clear cache to get those changes to register on the site (especially when logged out).

For more about custom templates read this Electric Citizen Blog Post.

Template Naming Conventions & File Locations

Drupal Template Naming— Drupal templates follow certain naming conventions. Each template starts with the entity type. A block template will start with block while a field template will start with field and so on and so forth. Every single Drupal entity has a template associated with it.

Template names go from the global entity level to the specific instance of that entity. The block.html.twig affects all blocks on the site while block--menu-block.html.twig would only affect all menu blocks on the site. More specifically, block--menu-block--footer.html.twig would affect only the menu block for the footer menu. File name entity levels are always separated by two hyphens. The words in the machine name of an entity level (example: main-menu) are always separate by a single hyphen.

Unless you know that the custom template you are creating is something that will be used at a high level (adding block name classes to all blocks for theming purposes), name your template as specifically as possible in order to prevent the customization from spilling over into other instances of that entity later on. You would never want to use something like block--basic.html.twig to add some special HTML structure to a basic block you created to add social media links in the footer because that code would be inherited by any other basic blocks created on the site.

Available template names are provided by green theme suggestions seen when using a browser inspector if theme debugging is turned on. If you do not see a template name suggestion that fits the template you need to create, you can add more through theme suggestions hook functions in the citizen_patterns.theme file.

Custom Drupal templates must always go in the /templates folder because a template can only get inherited into a template below it in the file structure. Having all custom templates in zzz-custom means that templates can be inherited from both Pattern Lab (using the Drupal Component Libraries module) and from the default Stable templates in /templates.

Twig Overview

Drupal 8 and Pattern Lab both use Twig templates. Twig is a PHP templating engine which is part of the Symfony framework. Twig is sort of like the Sass of PHP.

Twig has thee main delimiters that are used to build code. You can write expressions (variable outputs, strings, Twig functions, etc) using {{ }}. You can execute statements with tags such as if, for (loops) and set (define variable) using {% %}. Comments are adding using {# #}. Whenever using a Twig delimiter in a template, it is preferable to separate the opening and closing part of the delimiter from the content using a space on each side: {{ content.field_background_colors.0.value|clean_class }} or {% if content.field_background_colors.0.value|clean_class == "white" %} is-white {% endif %}

Twig templates should be commented use descriptive comments ({# My great Twig block does this... #}) the same as you would JS or Scss to explain your code to other developers (or future you from 6 months from now). Template name comments are not needed since templates are not compiled into single files like Scss. Component section name comments are recommended when using Twig blocks.

Contrib modules like Twig Teak and Twig Extensions help expand the native Drupal Twig functions. Twig Tweak is an OC Patterns dependency and used to insert media entities with the drupal_entity() function and Drupal Blocks with drupal_block(). Twig tweak is also what proves the dd() variable dump function used in custom templating.

Twig Extensions is used on as needed basis to introduce text and array functions and filters like |truncate and |shuffle.
