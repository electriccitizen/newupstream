Atomic Design Principles

Citizen Patterns is an uses an atomic design approach to theming. Atomic design is a system for developing a component-based design from the smallest element to the largest.

The Atomic Order:

  • Atoms
  • Molecules
  • Organisms
  • Templates
  • Pages

This theme defines global elements like color, fonts, macros and mixins, and builds them into text, links, icons, buttons, etc. These elements are built into blocks and menus. More advanced elements are widgets and the site header and footer. Everything is combined at end into layouts, pages and nodes.

Pattern Lab File Types

Pattern Lab components are made up of several file types:

  • SCSS
  • JS
  • TWIG
  • YML
  • MD

SCSS— SCSS files hold the SASS theming code that gets compiled into CSS for the site. SCSS files should never be visible to Pattern Lab and thus should always be preceded by an _. Each component generally gets its own .scss file even if it doesn't need any of the other file types. Some .scss files hold because they theme at a slightly higher level (example: _widgets.scss).

JS— JS files hold th jQuery scripts for the site. Most JS files should be hidden from Pattern Lab unless they are needed to test/demonstration specific component functionality. NOTE: Even if a JS file is visible to Pattern Lab, it will not be functional. It must also be included in theme/components/_meta/_01-foot.twig, assigned to a library in the theme's .libraries.yml and attached in a Twig template.

JS files are attached to Twig templates (both in Drupal and Pattern Lab) through the {{ attach_library() }} Twig function. When a JS file is assigned to a library, it should also have its specific dependencies assigned.

TWIG— Twig templates form the backbone of both both Drupal and Pattern Labs theming structure. Pattern Lab uses pure .twig templates while Drupal uses .html.twig templates. Twig is a PHP-based templating engine that compiles code down to PHP. Think of it like the Sass of PHP.

YML— YML files are commonly used for configuration purposes. Pattern Lab uses it as structure that defines strings and arrays of demonstration content to be transmitted to Twig templates as variables. These variables can then be inherited into Drupal and replaced with Drupal specific content.

Some common YML syntax:

  • Indentation: Two spaces indented start a sub level of structure. YML does not accept tabs for indentation.
  • Comments: # is used for comment lines.
  • List items: Can be denoted either by a leading hyphen – or enclosed in square brackets [] and separated by a comma.
  • Array items: Arrays are represented using a semi-colon:. Their key values are either set per indented line after the array variable, or are enclosed in curly brackets {}.
  • Strings: Strings are normally unquoted but can be enclosed in single or double quotes. Double quotes allow special characters within the string to be escaped using a backslash \.

MD— MD files are text files created in markdown language and used plain text documentation. OC Patterns specifically uses them for component documentation that is shown in Pattern Lab only. Adding documentation in a .md file ensures that the documentation does not get rendered into Drupal either as file comments or as blocks that need to be replaced with empty content in Drupal.

Hidden Files in Pattern Lab

By default, all files of any type are visible to Pattern Lab. Each file's type will defined how it is used (as specified in the File Types documentation section). However, some files should not be visible to Pattern Lab. Files can be hidden from Pattern Lab by using a leading underscore _ (example: _styleguide-specific.scss).

As previously specified, Scss files should NEVER be visible to Pattern Lab.

JS files should be hidden unless they are specifically needed for Pattern Lab testing/demonstration. JS files visible to Pattern Lab must also be called in the theme/components/_meta/_01-foot.twig file.

YML files should always be visible to Pattern Lab as their configuration is necessary for their corresponding Twig files.

MD files should also always be visible to Pattern Lab because their very purpose is to document Pattern Lab components.
