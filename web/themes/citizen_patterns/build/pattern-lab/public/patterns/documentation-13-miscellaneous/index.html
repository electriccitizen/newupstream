Dev Specific Configuration

Electric Citizen uses a full-site configuration export workflow (Continuous Integration). This means that ANY configuration changes on your local site (except those specifically limited to local environments) is pushed to the development environments when you export you work.

However, some configuration is tied to content—which is not exported. Paragraph file icons and custom blocks used in block layout and Layout Builder are prime examples. If you create a new custom block and add it to the footer locally then export your configuration, your block will show as missing or broken on the dev site—because the block does no exist there. In these cases, it is necessary to some configuration on dev (or whatever the highest level of environment is currently). When doing non-local configuration work, make sure that you alert all developers who may be working on the site before you start so that they do not push work that will override your dev changes before you get them into the workflow.

Paragraph Icon dev config— Paragraph icons need to be added in two places: a) node.css in the Citizen module (for the add buttons); b) the edit for of the paragraph bundle itself (for the small icon shown when the paragraph is closed on the node editing form). This second icon needs to be added to both local and dev simultaneously and then your local config should be exported to dev.

Custom Block dev config— Custom blocks should be created and placed where they are needed directly on dev (or whatever the current highest level environment is. Dev should then be imported to your local (through site:refresh or whatever the current workflow command is to sync your local database and code with dev) and re-exported back up to dev. This gets the correct block ID in assigned to the block in both environments and gets the configuration into the workflow stream.

Working with Git

Electric Citizen uses a Git workflow to run our Continuous Integrations. You can work with Git either directly through the command line, or with a Git Client software application like Tower or Kraken. NOTE: We have found better Drupal 8 git results with Tower over Kraken.

You should push your work to git early and often. The more frequent your pushes and the smaller the changes, the easier it is to revert them if its ever needed. Changes from other developers should be pulled in as soon as the become available.

Personal Task branches— All of your work should be done on personal task branches—never in the develop or master repo branches. Once your work is pushed and approved to go the develop environment, your branch is merged into develop (or master) and then deleted. NOTE: After each task is complete and merged into the main repo, you should refresh your local site from the canonical site and pull any changes from other developers. Your next task will then start on a new branch.

Commit Comments— Git commits should have descriptive comments that let another developer know what changes/additions/removals you made. When possible, include a ticket number with your commit message.

Onboarding/Workflow readmes— All Electric Citizen Drupal 8 sites (with the exception of Electriccitizen.com) have readme docs directly in the repo that will provide specific instructions to spin up a local copy of that site and the workflow needed to push/pull.
