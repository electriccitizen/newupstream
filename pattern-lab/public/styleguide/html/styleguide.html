  • One Password password manager (recommended)

Rules of the Road

Push early and often— All EC D8 sites use a GitHub based workflow. Pushing early in a project and pushing frequently will help prevent you from falling behind other developer's working on the same repo and will make a rollback to an earlier commit easier if necessary.

Theme mobile first— All page mockups come with a mobile version (and often a tablet sized version as well). It is almost always best and easiest to theme from the mobile version to the desktop version as most of the theme breakpoints run from small to large. Mobile versions are also usually less intricate than desktop so you can theme additively rather than subtractively.

Bite-sized theming— EC follows a component (atomic) approach to theming. We always try to theme the smallest chunks first and them put them together to complete the overall design. This helps ensure consistency while limiting repetition.

Everyone loves Chatty Cathy— All of your Sass, Twig and JS awesomeness might make perfect sense in your head while building, but for another developer (or even future you from 6 months from now) it can often be very painful and time consuming to pick apart what you were thinking. Comments are the best way to clear the path to easy and efficient development. The styles of commenting used by EC in each type file are covered in the best practice section for each type of theming.

Use all the things— Designs usually come with both .ai files and .jps. Its often most helpful to get all the sizes, colors and overall feel from the jpgs using Photoshop and then to do the type sizes using the Illustrator files.

pattern lab

▼

Pattern Lab Overview

Pattern Lab is a tool for creating atomic-based style guides for websites. Citizen Patterns uses Pattern Lab as its Drupal theme engine for template inheritance, Sass & JS compiling and other theme tasks, but Citizen Patterns is not meant to be a style guide that is passed on the to the client. At this time it is a tool for themer's only and as such is not deployed to git or viewed outside of a local development atmosphere.

Citizen Patterns is based on the Emulsify theme developed by Four Kitchens and as such we do not have 100% control over all node module aspects of the theme, but very little of the original Emulsify theme implementation is still left.

running the theme

npm start

▼

NPM & Node.js

NPM is used to install small packages of code that run site tasks. These are very small configuration scripts defined in a package.json file and run through the command line. The big advantage of npm scripts is that you are always up to date. If a package is updated, the run script will automatically use the updated version (unless specified otherwise).

NPM modules are defined in the package.json as dependencies and are fetched and installed by Node.js, which is a javascript based run-time builder. NPM Scripts in the package.json define tasks for those node modules.

Task Runners

Gulp and Grunt are common task runners. that function very similar to NPM scripts and either can be used on its own, or in combination with NPM scripts to run theme tasks.

Citizen Patterns takes advantage of both NPM and Gulp in order to power the theme tasks. Gulp allows use to specify unique tasks that expand npm modules installed by the theme. CRunning those tasks from a single npm command allows us to concurrently run different tasks without having to enter and re-enter multiple Gulp commands.

Citizen Patterns NPM Start

After install (install instructions in theme ReadMe), Citizen Patterns is run using a single command line command: npm start run in the theme root folder. NPM Start kicks off these Gulp tasks:

  1. Gulp Clean
  2. Gulp Compile
  3. Gulp Theme

These Gulp tasks are set in the Emulsify-Gulp gulp task collection fetched as a node module from the Four Kitchen's Emulsify Gulp repo when the theme is first installed. All tasks set in the default Emulsify-Gulp repo index.js file which constructs tasks from the repo's gulp-config file that can be overridden or disabled in the local.gulp-config.js file in the theme root. New gulp tasks can also be set in the local-config file and added via the npm start command.

Gulp Clean— Gulp Clean runs the Scss and ES Lint linting.

Gulp Compile— Gulp Compile finds the individual icons in theme/images/icons/src and compiles them into a single sprite then builds selector classes and mixins that set the proper background position for each icon. Icon base size (80px x 80px)is set in the local.gulp-config override for this task.

Gulp Theme— Gulp Theme calls the Gulp Serve task defined in the Emulsify Gulp repo's index.js file which runs the Sass & JS Compiler's, runs the Sass to Yml conversion (for colors), watches all the pattern lab files for changes and runs the browser sync, and finally builds the local pattern lab.

Emulsify Gulp also has options for Web Speed Tests and Build/Deploy functions for GH Pages, but OC Patterns is not using those at this time.

adding & removing pl files

▼

Adding New Pattern Lab (PL) Files

New files of any type can be added to any level of Pattern Lab—although the index numbers might need to be rearranged for some sections. After a new file is added, if you are currently running the theme (npm start is active in a terminal window), depending on the type of file you are adding, you may need to quit the current theming command run (control + C in terminal) and start a new one with npm start. It is a good practice to get into doing this whether you have to for the file type you are adding or not. File names should use - instead of _ to denote spaces unless there is a specific need for _ (like _icon_sprite.scss) which is generated by a gulp task. No Pattern Lab visible files can be named exactly the same as another, and index numbers do not count in the name. If two files have the same name (even with different indexes) the second one will not show in Pattern Lab.

New Icon SVG Files— If you are adding a new svg file to the icon sprite via theme/images/icons/src, you will need to restart the theme to add the icon to the sprite. NOTE: You will also need to recreate the viewbox position x/y values for any svg tag's in twig files (cough:: Social Media widget icons) because icons are repositioned in the sprite each time it is re-generated.

Icons used as backgrounds will automatically reposition themselves to work unless the new icon(s) you added created another column or row in the sprite. Easiest way to tell is to look at the icons grid (Atoms > Images > Icons) to see if everything looks correct. If the size or placement is off, load the sprite (found in theme/dist/svg) into a browser or Adobe Illustrator and count the rows/columns and change the values for those variables at the top of Base > _03-background-sizes.scss to the new number of rows/columns. The background svg's will then fix themselves.

Adding Hidden Files— By default, all files or all types (twig, sass, yml, js, md) added to Pattern Lab will be used/shown in Pattern lab. However, there are some files you do not want to use/show in Pattern lab—either because they are being only used in Drupal or are being compiled/inherited into another file. These files can hidden from Pattern Lab by including a leading _ in their file name. Scss files are a great example of this. No .scss files should be shown in Pattern Lab because they are all compiled into the css which is shown in Pattern Lab. Most .js files should also be excluded because they are primarily used in Drupal, but should be kept with their Pattern lab components.

File Index Numbers— Many Pattern Lab files are preceded by an index number. This number overrides the default alphabetical file name structure to force a particular order on files. This is very useful for forcing an atomic design approach to theme structure as well as ensuring Sass mixin inclusion.

New Twig Files— New Twig files and any twig files changes that are inherited into Drupal html.twig files necessitate a site cache clear after creation/alteration in order to get the changes to show in Drupal. The same is true of changes made via macros.

Citizen Patterns is an uses an atomic design approach to theming. Atomic design is a system for developing a component-based design from the smallest element to the largest.

file types

▼

Pattern Lab File Types

Pattern Lab components are made up of several file types:

  • SCSS
  • JS
  • TWIG
  • YML
  • MD

SCSS— SCSS files hold the SASS theming code that gets compiled into CSS for the site. SCSS files should never be visible to Pattern Lab and thus should always be preceded by an _. Each component generally gets its own .scss file even if it doesn't need any of the other file types. Some .scss files hold because they theme at a slightly higher level (example: _widgets.scss).

JS— JS files hold th jQuery scripts for the site. Most JS files should be hidden from Pattern Lab unless they are needed to test/demonstration specific component functionality. NOTE: Even if a JS file is visible to Pattern Lab, it will not be functional. It must also be included in theme/components/_meta/_01-foot.twig, assigned to a library in the theme's .libraries.yml and attached in a Twig template.

JS files are attached to Twig templates (both in Drupal and Pattern Lab) through the {{ attach_library() }} Twig function. When a JS file is assigned to a library, it should also have its specific dependencies assigned.

TWIG— Twig templates form the backbone of both both Drupal and Pattern Labs theming structure. Pattern Lab uses pure .twig templates while Drupal uses .html.twig templates. Twig is a PHP-based templating engine that compiles code down to PHP. Think of it like the Sass of PHP.

YML— YML files are commonly used for configuration purposes. Pattern Lab uses it as structure that defines strings and arrays of demonstration content to be transmitted to Twig templates as variables. These variables can then be inherited into Drupal and replaced with Drupal specific content.

Some common YML syntax:

  • Indentation: Two spaces indented start a sub level of structure. YML does not accept tabs for indentation.
  • Comments: # is used for comment lines.
  • List items: Can be denoted either by a leading hyphen – or enclosed in square brackets [] and separated by a comma.
  • Array items: Arrays are represented using a semi-colon:. Their key values are either set per indented line after the array variable, or are enclosed in curly brackets {}.
  • Strings: Strings are normally unquoted but can be enclosed in single or double quotes. Double quotes allow special characters within the string to be escaped using a backslash \.

MD— MD files are text files created in markdown language and used plain text documentation. OC Patterns specifically uses them for component documentation that is shown in Pattern Lab only. Adding documentation in a .md file ensures that the documentation does not get rendered into Drupal either as file comments or as blocks that need to be replaced with empty content in Drupal.

pattern lab hidden files

▼

Hidden Files in Pattern Lab

By default, all files of any type are visible to Pattern Lab. Each file's type will defined how it is used (as specified in the File Types documentation section). However, some files should not be visible to Pattern Lab. Files can be hidden from Pattern Lab by using a leading underscore _ (example: _styleguide-specific.scss).

As previously specified, Scss files should NEVER be visible to Pattern Lab.

JS files should be hidden unless they are specifically needed for Pattern Lab testing/demonstration. JS files visible to Pattern Lab must also be called in the theme/components/_meta/_01-foot.twig file.

YML files should always be visible to Pattern Lab as their configuration is necessary for their corresponding Twig files.

MD files should also always be visible to Pattern Lab because their very purpose is to document Pattern Lab components.

Citizen Patterns has a background sizes file in the base folder (00-base/_03-background-sizes.scss). This file is used as an automatic background sizer for SVG icons used as backgrounds. This corrals all the background svgs into one area and necessitates only a few variable changes when a new icon is added rather than each instance of an svg background needing to be updated each time the sprite is generated.

Changing these values when a new row of column is added while automatically update all background SVG icons to use the proper positioning.

@include backgroundSize($columns * $hamburger,$rows * $hamburger);

Mixins & Placeholders

A lot of the power of Sass comes from mixins and placeholders. These are bits of reusable code that prevent you from having to type the same thing over and over. Mixins are denoted as @mixin example{ my scss code here } and called with includes: @include example;. Placeholder are denoted as %example{ my scss code here} and called with extends: @extend %example.

The actual difference between Mixins and Placeholders can be tricky but here are a couple of key points:

  1. Variable arguments cannot be passed through placeholders.
  2. Placeholders must be used in the file they are declared in, or that file must be imported into the file they will be used in.
  3. Mixins are printed for each selector instance, but place holders are printed once with the selectors using them grouped together.
The top two key differences definitely make the argument for using nothing but mixins, however, as noted in the last key difference they do slightly inflate the size of your stylesheet versus placeholders.

Citizen Patterns is generally favors mixins over placeholders because of the allowance of variable arguments, but placeholders are definitely recommended for static Scss mixins with no contextual needs.

Citizen Mixins

Most mixins are defined in 00-base/_02-mixins.scss. Background mixins are defined in 00-base/_03_background-sizes.scss and breakpoint mixins are set in 00-base/_00-breakpoints.scss. Some other more specific mixins are set in their respective component Scss files (example: @mixin blockTitle{} set in 01-atoms/02-text/00-headings/_headings.scss).

All mixins set in the theme should be listed as a mixin-tile in 00-base/global/04-mixins/mixins.twig so that there is a visual Pattern Lab reference for them. The mixin tile should give the mixin name, the @include call for it, a short description, any argument values needed, and a list of similar mixins (similar mixins do not need to have a tile for each).

A new mixin should be created anytime a piece of Scss is being reused in multiple files of the theme and there are contextual variables being set in each instance.
Example:
@mixin columns($columns, $gap){
-webkit-column-count: #{$columns};
-moz-column-count: #{$columns};
column-count: #{$columns};
-webkit-column-gap: #{$gap};
-moz-column-gap: #{$gap};
column-gap: #{$gap};
}

Most placeholders are defined in 00-base/_01-placeholders.scss (which is imported into other Scss files as needed. File specific placeholders can be defined within the file they are being used in.

A placeholder should be created for any common piece of Scss that is reused across the site (or within a single file) but no contextual arguments are needed.
Example:
%listReset{

padding: 0;
margin: 0;

sass best practices

Citizen Patterns Sass Best Practices & Conventions

  1. Comments— File names are commented at the start of Scss files.
    /*===================
    MY GREAT FILE NAME
    ========================*/

    Major sections/groups of Scss code are commented within the file.
    /* --------------------------
    Section/Group name
    --------------------------*/

    Single Scss elements withing sections are commented within each section.
    /* Element Name */

    Descriptive comments are added within each element above what they are describing.
    // my great descriptive comment

    Closing comments are added to Scss element closing brackets using a description comment on the same line as the closing bracket.
    }//end .my-selector

    Closing comments are also used when nest levels get more than 2 or 3 deep or it is not easy tell what a nested closing bracket pertains to due to the amount of nested code.

  2. Nesting— Top level selectors generally pertain to the component of each Scss file. Example: In 03-organisms/00-paragraphs/button/_button.scss all Scss code is nested within the .paragraph--type--buttons{} selector. because that Scss file pertains to that specific component.

    For files with multiple components (like _widgets.scss or _views.scss), top level selectors would generally be each main component in that file (each view instance would be its own top level selector).

    Selectors should never be nested more than 5-6 levels deep and 2-4 is usually preferred for clarity. Example: A view selector would have the view content, views row, and each view field all as second level nested elements—rather than a view selector with view content nested then views row nested inside that, with view fields nested inside the row. Using shallower nesting techniques makes the view easier to read at a glance and allows for clearer and more specific breakpoint nesting.

  3. Breakpoint nesting— Citizen Patterns uses breakpoint mixins based on grid variables set in 00-base/00-breakpoints.scss for responsive theming. Each individual selector (nested or not) should have its breakpoint styles nested directly in the selector. Example: .views-row{ default styles for the row; @include tab{ tablet styles for the row;} } and .my-viewfield{ default styles for the field; @include tab{ my tablet styles for the field;} }.

    Nesting breakpoint theming directly in the selector it pertains two provides an easier to read Scss file and limits the amount of selector repetition. Breakpoint theming should be at the end of each selector and should go in break point order: default, tab, desk, contentStop, etc. Special breakpoints can also be used to limit theming to just phones (less than 500px) or just mobile menu widths for example.

  4. Grid theming— Citizen Patterns does not use any defined grid system like Singularity or CSS Grids. This is because needs of the design and the build does not often make a single simple system worthwhile since things move position between mobile and desktop or are grouped in complex ways. Instead we use flex theming whenever possible and floats if absolutely necessary.

    Whenever possible use flex theming with the custom flex mixins defined in 00-base/_02-mixins. @include safeFlex($direction,$wrap,$justify,$align); will provide all necessary arguments for any type of flex theming you need for layout elements and will ensure it works back to IE10. @include safeFlexOrder($order) allows you to set the flex order of a child element with a flex layout and @include safeFlexSize($grow,$shrink,$size) lets you set flex sizing on the child element (regular percentage, vw, or pixel sizing is usually used instead of safeFlexSize however).

  5. Mixin, Mixin, Mixin— Use mixins and placeholders whenever you find yourself repeating the same Scss code in more than a few spots. Mixins with arguments can definitely help your efficiency while placeholders help keep the CSS lean. View defined mixins and placeholders in the Base > Global > Mixins Reference sheet. Whenever you create a new mixin, make sure you add a mixin tile for it in 00-base/04-mixins/mixins.twig so that it because visible in the mixins reference sheet.

    In order for a mixin to be used in a file, the mixin must be assigned in the file hierarchy before it is called in a file. This is why the 01-mixins.scss is one of the first top level files in the theme. Any file that uses placeholders that are defined in 01-placeholders.scss must import the 01-placeholders.scss file.

  6. Variables— All colors, grid sizes and fonts should be assigned to a variable. Mixins should take advantage of argument variables whenever their is a contextual need (like setting a different height for the vertRule mixin in different uses). If a mixin does not use an argument, it should probably be a placeholder.
  7. Camel Casing— When creating mixins, placeholders, or variables, always use camelCasing instead of hyphens or underscores.

JS in PL and Drupal

jQuery— jQuery is what we use to write 98% of the JS on a site. jQuery is a core Drupal dependency and all of the JS files in the theme are set up to use jQuery functions inside of Drupal behaviors.

When adding a new file, its usually easiest to copy an existing theme JS file and just delete everything between the opening line: (function($, Drupal) { and the closing line: })(jQuery, Drupal);. Those two lines let Drupal know that it should use its core jQuery file to execute the behaviors inside the file. If you do create a new file from scratch or adding new behaviors to a file, make sure that everything is between those lines.

Drupal Behaviors— All jQuery functions in theme JS files need to be inside of Drupal Behaviors. This ensures that all jQuery code gets called after the DOM has loaded so that you code is not jumping ahead of things it might need to function (like ajax). Drupal behaviors also comes with another line that attaches a context function. Using context ensures that your script is only searched within the given context rather than the entire document.

Example: Drupal.behaviors.slideshow = {
attach: function (context, settings) {
$(".paragraph--type--slideshow", context).once('slider').each(function(){ Drupal behaviors and contexts should never be repeated. ANy duplicate behavior will not be called in the DOM.

The above example opening lines of a behavior and context will search only instances of .paragraph--type--slideshow and will only run the code once for each instance it finds in the DOM. Using context helps make sure that you are not slowing down the site by searching the whole DOM for elements that are not there and by ensuring that code is only run once, rather than looping through each instance and running multiple times. This is especially important with .resize() responsive functions.

Limit JS as much as possible— While JS is a powerful tool, Twig & CSS expressions/animations can do a lot of what we used to need JS for. It is preferable to use Twig or CSS whenever possible to both limit the amount of files being called as well as proofing again people who have JS turned off in their browser and errors from external scripts/applications interfering with your scripts.

Component JS files are files that are attached to the Twig template files for the components they relate to. This done through a library attachment Twig function. Each component will generally have its own JS file and library—with some exceptions:

template inheritance

▼

Twig Template Inheritance

Twig template inheritance works both between Pattern Lab and Drupal and within each system itself. Any template can inherited into any other template as long as the template being inherited comes before the inheriting template in the hierarchy. This is why atomic design is so crucial. A global paragraphs.html.twig Drupal template can be extended (inherited) into all of the paragraph bundle (widget) specific templates so that the correct classes get picked up without having to be re-written each time.

Pattern Lab twig templates can be inherited directly into Drupal templates so that the component theming can be in Pattern Lab without even the need for a working Drupal site and surrounding HTML structure can be tailored as needed and displayed in both the style guide and Drupal.

Types of Inheritance— Templates can be inherited by another by using Extends, Include or Embed. They all function in a pretty similar manner, but there are some key differences:

  • Extends— Brings in the entire parent template into the child (inheriting) template. No content outside of the blocks defined in the parent template will be allowed in the child template. Parent blocks can have their content changed in the child template.
  • Include— Inherits the entire parent template into a specific place in the child template (rather than taking over the whole child template as extends does). The included template is rendered in the place in the child template where the inclusion happens. Included templates may have their variables changed by using with:{ } inside of the {% include "my-parent-template-name" %} call. Included templates may not have their block content structure changed (only variables).
  • Embed— Works as a sort of hybrid between Extends and Include. Parent templates are inherited into child templates at the place the embed happens. Variables can be changed by using with:{ } like include, but block content can also be replaced like with extend.
For more on template inheritance, read this amazing Electric Citizen blog post.

  2. When debugging, {{ kint() }}, {{ dd() }}, and {{ ksm() }} will get you all of the available variables to work with. {{ dd() }} is the fastest and generally best to use. {{ ksm() }} is used only in the .theme file.
  3. Templates for entities re-used site after site should be created in Pattern Lab and inherited into Drupal. Site specific templates can just exist in Drupal unless there is time in the project for a full site specific style guide.
  4. When creating custom templates in Pattern Lab, it is best to use Twig blocks around component sections. Even if the template is not currently going to inherited into anything else. Using blocks will future proof for later inheritance possibilities.

Custom Twig Templates

Every component in Drupal has a corresponding Twig template or even group of templates. Whenever you need to change or create the HTML structure of a component, need to attach a JS file to it for scripting, or want to add some special classes for theming, you will need to create a new custom template for the element or edit an existing template.

Existing Custom Templates— All existing Drupal custom templates for Citizen Patterns are stored in /templates/zzz-custom. Any other templates in the theme are default templates from the Stable base theme. If any of these templates need to be edited—it should be moved into one of the sub-folders in the zzz-custom folder. Make sure you move the whole template instead of just copying a version of it into the zzz-custom folder structure. NOTE: You may also inherit templates from outside zzz-custom into templates inside of zzz-custom since it is at the end of the /template file hierarchy.

Existing Pattern Lab templates are all in the /components/_patterns file structure. Any new templates added to Pattern Lab should be added there in the appropriate component area. If the new Pattern Lab template is going to be inherited into Drupal, a custom template must also be created (or edited) for the inheritance in /templates/zzz-custom.

Any Twig template found outside of the Citizen Patterns theme is either a Drupal core template or one provided by a contributed module. Whenever possible, it is best not to customize these templates because this may cause errors after future core/module updates, but if it is necessary to do so, you should copy the template (not move it like with customizing Citizen Pattern default templates) into the zzz-custom folder structure.

Whenever you create a new template or rename a template, you must clear cache in order for the new template to be read by Drupal or Pattern Lab. Often when you make changes to a template, you may also need to clear cache to get those changes to register on the site (especially when logged out).

naming conventions & file location

▼

Template Naming Conventions & File Locations

Drupal Template Naming— Drupal templates follow certain naming conventions. Each template starts with the entity type. A block template will start with block while a field template will start with field and so on and so forth. Every single Drupal entity has a template associated with it.

Template names go from the global entity level to the specific instance of that entity. The block.html.twig affects all blocks on the site while block--menu-block.html.twig would only affect all menu blocks on the site. More specifically, block--menu-block--footer.html.twig would affect only the menu block for the footer menu. File name entity levels are always separated by two hyphens. The words in the machine name of an entity level (example: main-menu) are always separate by a single hyphen.

Unless you know that the custom template you are creating is something that will be used at a high level (adding block name classes to all blocks for theming purposes), name your template as specifically as possible in order to prevent the customization from spilling over into other instances of that entity later on. You would never want to use something like block--basic.html.twig to add some special HTML structure to a basic block you created to add social media links in the footer because that code would be inherited by any other basic blocks created on the site.

Available template names are provided by green theme suggestions seen when using a browser inspector if theme debugging is turned on. If you do not see a template name suggestion that fits the template you need to create, you can add more through theme suggestions hook functions in the citizen_patterns.theme file.

Custom Drupal templates must always go in the /templates folder because a template can only get inherited into a template below it in the file structure. Having all custom templates in zzz-custom means that templates can be inherited from both Pattern Lab (using the Drupal Component Libraries module) and from the default Stable templates in /templates.

twig overview

▼

Twig Overview

Drupal 8 and Pattern Lab both use Twig templates. Twig is a PHP templating engine which is part of the Symfony framework. Twig is sort of like the Sass of PHP.

Twig has thee main delimiters that are used to build code. You can write expressions (variable outputs, strings, Twig functions, etc) using {{ }}. You can execute statements with tags such as if, for (loops) and set (define variable) using {% %}. Comments are adding using {# #}. Whenever using a Twig delimiter in a template, it is preferable to separate the opening and closing part of the delimiter from the content using a space on each side: {{ content.field_background_colors.0.value|clean_class }} or {% if content.field_background_colors.0.value|clean_class == "white" %} is-white {% endif %}

Twig templates should be commented use descriptive comments ({# My great Twig block does this... #}) the same as you would JS or Scss to explain your code to other developers (or future you from 6 months from now). Template name comments are not needed since templates are not compiled into single files like Scss. Component section name comments are recommended when using Twig blocks.

Contrib modules like Twig Teak and Twig Extensions help expand the native Drupal Twig functions. Twig Tweak is an OC Patterns dependency and used to insert media entities with the drupal_entity() function and Drupal Blocks with drupal_block(). Twig tweak is also what proves the dd() variable dump function used in custom templating.

Twig Extensions is used on as needed basis to introduce text and array functions and filters like |truncate and |shuffle.

site specific colors & fonts

Site Specific Colors

Citizen Patterns has some base colors in place that are needed for Pattern Lab theming and testing in between Drupal site spin ups. Obviously, these need to get changed for each site.

Site Colors— Site colors come from the Design Direction sheet provided by the design and from mock-ups as they become finalized. All colors are entered in the 00-base/global/01-colors/_color-vars.scss file as variables. There should never be a color used on the site that is not a variable. When a color is added to (or changed in) _color-vars.scss as a variable, it will automatically be picked up by Gulp and added to the colors.yml file which displays the color reference palette in Pattern Lab.

Colors should stick with non-fancy color names with light/mid/dark options added as needed ($red and $redDark for instance). If you change one of the base colors to use on a specific site (like $grey), move that variable into the /* Client Colors */ section of _colors-vars.scss. It's not really necessary but its good housing keeping and makes all the site-specific colors easier to find later.

Some colors use the non-color names but change every site:

  • $siteBack == the site background color
  • $borderRule == global color for all borders and rules
  • $link == global link color
  • $linkHover,$linkActive == global colors for link states
  • $buttonBack,$buttonbackHover == global default button colors
These colors should not be moved or renamed—just change the hex colors as needed.

Site Specific Fonts

Citizen Patterns loads Open Sans from Google Fonts as the default font for base theming. This font will most likely need to be replaced on a per site basis.

Loading Fonts— Depending on where you are getting the site fonts from, they can be loaded in a few different ways:

  1. Using the Fonts library: Open citizen_patterns.libraries.yml and scroll all the way to end. You'll see the fonts: library with Open Sans being loaded as a css external sheet from fonts.googleapis.com. If the fonts you are loading use a link rel="stylesheet" header call, you can just replace the Open Sans url with the new font url.

    If you need a js script tag in the header to load the fonts, you can add js: to the fonts: library and paste in the url to fonts. Just make sure you have { type: external } after it. Add minified: true after the external notation if you are using a minified font sheet.


  2. If either of these don't work for some reason, you can add the entire head tag needed for the font into the {% block headInsert %} placeholder in html.html.twig in /template/zzz-custom/layouts.

  3. Hosting fonts directly on the site is also possible. Add the font files into the /fonts folder in theme. Add @font-face css at the start of 00-base/global/02-fonts/_fonts-vars.scss:

    Example:
    @font-face {
    font-family: 'myFont';
    src: url('../fonts/myfont.eot');
    src: url('../fonts/myfont.eot?#iefix') format('embedded-opentype'),
    url('../fonts/myfont.woff') format('woff'),
    url('../fonts/myfont.ttf') format('truetype'),
    url('../fonts/myfont.svg#myfont') format('svg');
    font-weight: normal;
    font-style: normal;
    }

If you used one of the first two methods to load the fonts, you can test to see if they are loading by inspecting any page of the site in a browser and checking the head and foot tags to see if you link and script tags are there.

Font Variables— Once the fonts are loading on the site (or if you used the third method), replace the font variables in _fonts-vars.scss with the new font-families to use them in Scss. Heading sizes and colors are updated in 01-atoms/text/00-headings/_headings.scss but they always use the $header font-family variable (except h1 which may use $displayHeader). Text styles are updated in 01-atoms/02-text/text/_text.scss.

CKE Editor WYSIWYG font– It is a good practice to match styles in WYSIWYG fields as closely to the rendered results as possible to help editors visualize how they are building the page. To that end, when new fonts are added, the fonts for CKE editor should be updated as well. You can load the fonts in citizen_patterns.info.yml under ckeditor_stylesheets: the same way you would load the regular fonts—except you don't need to specify external. Then update the necessary styles in /components/css/cke.css to reflect the new fonts and header styles.

new elements

▼

New Elements

Every new site with come with needs for new elements in order to build the design. The goal is to have pretty much every element on the page easily editable by a client without allowing them to break the design. For that, we rely heavily on paragraphs and block types—rather than creating non-editable content in Twig templates, blocks, or WYSIWYG HTML.

Creating New Elements— Just about every new element you create (other than content types) will need a Twig template so that you can create the best HTML structure will still using variables so that editors can control the content. All of these templates will go into the applicable folder in /templates/zzz-custom. Not all of these templates need to created in Pattern Lab first then inherited into Drupal. Usually that is only done with elements that will be part of the base build (unless the project allows time for a full site-specific style guide).

New Scss Files— Pretty much every element should have a new Scss file to itself—even if it doesn't need a template. Depending on the element type, you'll create a Pattern Lab folder and an Scss file for it in the following locations:

  • Block: 02-molecules/blocks
  • View: 03-organisms/01-views
  • Paragraph (widget): 03-organisms/00-paragraphs
  • Layout: 04-pages/layouts
  • Content Type: 04-pages/nodes
When creating new Scss templates—you will need to restart Pattern Lab to register the new template.

Blocks

New blocks should created on an as needed basis for elements with content that is repeated throughout the site. Example: A photo, text and button element is used as a call to action above the footer in multiple areas of the site. This would be a new custom block type with a block created for that element which could them be inserted where needed using the block placer widget.

Custom Block Types— Custom block types in DB are fieldable entities that are re-usable through the site. They can be added to pages either through the Block Layout UI, Panelizer, or the block placer paragraph widget. When a new custom block type is created, it must be added to the allowed blocks in the block placer paragraph bundle block field in order for it to be used by editors.

Basic Blocks— Basic blocks are the default blocks in Drupal that allow just a description (block title) and a body field. These are generally just used simple bits of repeatable text.

Content Types

Every site will need content types to hold the client's custom content. New content types will be defined by the project manager.

Creating new content types— You can create new content types by going to Structure > Content Types in Drupal and adding a new one. Enter a name and description and choose your settings:

  • Submission: Leave as is
  • Publishing Options: Only use Published and Create New Revision
  • Display: Do not display author and date
  • Add Another: Leave as is
  • Menu: Add to main menu only if directed by the Project Manager.
Add your fields in manage fields and set the display as needed. If the content type is going to be displayed to an end user, it should always use Layout Builder. You may need to create a new layout for it, depending on design.

New content types that are going to be shown to an end user also need to have URL Pattern set. This can be created in Configuration > URL Aliases > Patterns. Add a new pattern and set it following the existing conventions. Usually the pattern is content-type-name/[node:title]) unless it is in a menu. Then use [node:menu-link:parents:join-path]/[node:title].

New content types also need to have their editing permissions set in People > Permissions. Follow the existing conventions for nodes—unless directed otherwise.

Unused default content types— Not every site needs all of the default Citizen Patterns content types. Best practice is not to delete them (as some are tied into the site through dependencies so heavily that you can crash the site by deleting them). The client may also later want one they did not originally need. Instead just turn off non-admin editing permissions for all unneeded content types. No client will ever have administrator roles so they will never see them.

Paragraphs (widgets)

The bulk of new elements created for a site are paragraphs (interchangeably called widgets). Some paragraphs may even be created that an editor will never use solely to ease the life of the site builder (such as View Placer).

New Widgets— New widgets can be created at Structure > Paragraph types. Add the fields you need (pretty much every widget uses a Widget Title field and an Admin description field), set the form display and display settings you need. NOTE: you should always create a Preview display that shows only the Admin description and any special styling choices.

New paragraph types also need to have their permissions set in People > permissions. All roles use the View Content permission and Admins, Senior Editors, and Editors generally all can Create, Edit and Delete each widget—unless it is for site building use only (then just admins get these permissions).

Next, each new paragraph gets added the OC Widgets field on the content types it will be used in.

Paragraph templates— Generally, all new paragraphs will get their own custom template in /templates/zzz-custom/widgets. This is so that the HTML structure can be optimized and any special styling keys (select list or boolean fields) can be added as classes for theming. All new paragraphs templates should inherit the paragraph.html.twig base template. This ensures that they get the proper flex, type, and id classes needed for theming. All new paragraph content in the template then goes inside the {% block widgetContent %} block. Most new paragraphs don't need a Pattern Lab template unless there is time in the project for a full site-specific style guide.

Paragraph Scss & JS files— All new paragraphs should have their own folder and Scss file Pattern Lab. This file goes in 03-organisms/00-paragraphs. Any specific js file would also go in this folder and would be attached as a library in {% block widgetContent %} in the paragraph template.

Image Styles & Responsive Media

Many designs will need new image styles. These can be created in Configuration > Image Styles.

New Image Styles— New image styles should be named something specific to their size, aspect ratio, or specific place in which they will be used. If the new image style is one that will be cropped, it should be cropped using the Focal Point Scale and Crop effect. If the image is a retina 2x size, it should also have and Image Style Quality effect of around 30-40%.

Responsive Image Styles— All small images (less than 600px) should also have a retina 2x style. This is a style exactly twice the size as the original with _2x at the end of the image style name. Every 2x style should have an Image Style Quality effect. Larger 2x styles (500-600px) should have a lower quality setting (30%) while lesser (100-200px) use a higher setting (40%). Reducing the quality significantly lowers the file size, but since it is retina and run through breakpoints, the visual difference is not noticeable.

Once the 2x style is created, a responsive image style needs to be created in Configuration > Responsive Image Styles. The responsive image style should be named the same as the original (1x) style and should always use the Open Citizen Patterns breakpoint group. Fallback image style is set to the original 1px style. Set the 1x and 2x styles for the number of breakpoints you need starting at mobile and going up to the size of the original image. (not all styles need all breakpoints). NOTE: You can also use different image styles for different breakpoints (example: medium 500 uses small 300 (1x and 2x) on mobile and medium 500 (1x and 2x) at tablet).

Breakpoints are set in citizen_patterns.breakpoints.yml and generally should not be edited or added to since they match the theme styling breakpoints. If you need a new breakpoint for some reason, talk to the lead developer first.

Media View Modes— After you create you image styles and responsive image styles, you need to create a Media View Mode in order to display the images on the site. View modes can be created in Structure > Display Modes > View Modes. The view mode should be named some similar to the image/responsive image style.

Media Bundles— Once the view mode for the new style is created, you must add it to the display settings in the Image Media Bundle. This is the media entity that controls how images are displayed on the site. You can added new fields here (example: media categories), change the form display for new images and define the view modes they can be displayed as. To use your new style on a field, create a new Custom Display by checking the box for your view mode on the Default tab of the image media bundle. Save, then you can edit your new display setting to display either the image style or the responsive image style you create. You can choose to display a caption (or other fields) or not. Most images do not display captions. NOTE: Documents and Video can have their display settings set in the same method (though they do not use image styles).

Display new image styles on a field— Once the display mode is set up in the image bundle, you can set the display settings for and image field to be: Rendered Entity > Your New Display. To set the new display in a custom template use the Twig Tweak function: {{ drupal_entity('media', content.field_citizen_image.0.target_id, 'my_new_style') }}.

Citizen Module

Working with Git

Electric Citizen uses a Git workflow to run our Continuous Integrations. You can work with Git either directly through the command line, or with a Git Client software application like Tower or Kraken. NOTE: We have found better Drupal 8 git results with Tower over Kraken.

You should push your work to git early and often. The more frequent your pushes and the smaller the changes, the easier it is to revert them if its ever needed. Changes from other developers should be pulled in as soon as the become available.

drush commands

Common Terminal Commands

Common D8 drush commands— Below are some common/useful drush commands for Drupal 8. NOTE: Always look at the readme file for each site repo to find site-specific import and export commands. All local drush commands must be followed by the site alias for that VM or preceded by 'lando' if on a lando VM.

  • drush cr all— Clears the site cache. Needs to be done when new files are added, Twig code is changed, new modules are added, or as first defense against fatal errors/warnings.
  • drush -y config-set system.performance css.preprocess 0 && drush -y config-set system.performance js.preprocess 0— Turns off the site CSS and JS file aggregation settings. This is usually done automatically during the site:refresh process.
  • drush sql-sync @pantheon.alias.dev @self— Fetches the database from the dev environment and syncs it to your local.
  • drush sql-dump --result-file=../database-dump.sql— Dumps you local database to an archive file to be shared with another developer.
  • drush pm-uninstall module_name— Uninstalls a module. NOTE: after being uninstalled module must be composer removed if no longer needed by the site. Modules uninstalled only on local environments but still used on dev/live environments do not get composer removed.
  • drush pm-enable module_name— Installs a module. NOTE: A module should be composer required before being installed (unless it is used on a different environment and is already required in the composer.json file).
  • drush upwd --password="admin" "admin" — Changes the admin (user 1) password to 'admin'. NOTE: This can also be done using drush uli.
  • drush config-delete module_name.settings— Fixes 'configuration already exists' module errors that sometimes happen when re-installing a module previously used on the site or upgrading a module. NOTE: This is a very rare case and if you are not entirely sure of what you are doing, the Technical Director or Lead Front End Developer should be consulted.

Common composer commands— Below are some common/useful composer commands for Drupal 8.

  • composer install— Installs all of the site dependencies listed in the composer.json file in the repo root. This is done when a site is first setup or is refreshed (done automatically on refresh).
  • composer update— Updates all of the site dependencies listed in the composer.json file in the repo root to their latest versions. NOTE: This should never be done globally using just the 'composer update' command. Instead this should be on a module by module basis as needed: composer update drupal/module_name. You can specify a specific version to update to by adding the version number at the end: composer update drupal/module_name:1x-dev.
  • composer require drupal/module_name— Adds a module and its dependencies to the site. You can specify a specific version to update to by adding the version number at the end: composer update drupal/module_name:1x-dev. NOTE: A module msut be enabled using either drush en module_name or through the Extends UI after it is required. You can also use composer require to add external git repos or other libraries as site dependencies if needed although it is preferred that this is done through npm in the theme package.json or through a CDN as an external library if possible.
  • composer remove drupal/module_name— Removes a module and its dependencies from the site. NOTE: Any module removed from the site must be first uninstalled using drush pm-uninstall module_name of through the Extend UI.

Troubleshooting

Citizen Patterns normally is pretty bulletproof if you are using the most recent versions of npm and node.js. If you do run into trouble when either installing the theme or running (npm start), follow these steps:

  1. Update to the most recent version of npm and node.js— Since this theme is based on Four Kitchens Emulsify it is best to run the most current versions of both npm and node.js in order to stay even with dependency needs.
  2. Make sure non of you files are misnamed or contain incorrect syntax— ES Lint (JS Linting) will cause the theme to quit running if it detects a JS error. It will give you the error and line to fix. New files (SCSS, JS & Twig) will also often cause the theme to quite running. Clear the site cache and re-run npm start. Twig syntax/variable errors will often cause a white screen with a fatal error message. use the Recent Log Messages report (Config > Reports) to get the specific Twig error, fix it and clear cache.
  3. Check the error to see if it has to do with theme dependencies— Sometimes theme dependencies will fail due to npm/Gulp developers falling behind or version conflicts. Often resolutions to dependency errors can be found in issue queues either in the Emulsify theme repo, the Emulsify Gulp repo, or on the npm package or dependency repo issue queue.
  4. Scream for help at the top of your lungs— When all else fails, the Lead Front-End developer and/or Technical Director can most likely find a solution to your problem.
